## Core Notes

### Two kinds of display, one shared question

Adding a screen to a project comes down to roughly two choices:

- **Character display (LCD1602):** shows only preset characters, in 16 columns × 2 rows. Simple, cheap, large glyphs.
- **Pixel display (SSD1306 OLED):** made of individual pixels (commonly 128×64), able to draw text, graphics and progress bars; self-lit, high contrast.

Either way, the core question is the same: **how does the data get into the screen?** The answer is a bus you've already met.

### LCD1602: characters and a parallel port

The LCD1602 has a built-in character table; send it a character code and it shows that glyph in the corresponding cell. The traditional connection is **parallel** — 4 or 8 data lines plus a few control lines, a lot of wiring.

Thankfully there's the **I²C backpack**: a small adapter soldered to the back of the screen that **squeezes a dozen wires into 4** (VCC, GND, SDA, SCL) and talks over I²C (see BUS2):

```cpp
#include <LiquidCrystal_I2C.h>
LiquidCrystal_I2C lcd(0x27, 16, 2);   // address is often 0x27 or 0x3F
void setup() {
  lcd.init(); lcd.backlight();
  lcd.setCursor(0, 0); lcd.print("Hello, Hardware");
  lcd.setCursor(0, 1); lcd.print("Line 2");
}
void loop() {}
```

### SSD1306 OLED: pixels and a frame buffer

The SSD1306 is a pixel display with an internal **frame buffer** — each pixel on screen maps to one bit in the buffer. Everything you "draw" in the library (text, lines, images) goes into that buffer first, then is pushed to the screen in one **display()** refresh. It usually runs over I²C (4 wires) too, address often 0x3C.

```cpp
#include <Adafruit_SSD1306.h>
Adafruit_SSD1306 oled(128, 64, &Wire);
void setup() {
  oled.begin(SSD1306_SWITCHCAPVCC, 0x3C);
  oled.clearDisplay();
  oled.setTextSize(1); oled.setTextColor(SSD1306_WHITE);
  oled.setCursor(0, 0); oled.println("Hello OLED");
  oled.display();               // push the buffer to the screen
}
void loop() {}
```

### The common pattern of display libraries

Almost every display library follows the same rhythm: **init → clear → set position/size/colour → write content → (pixel displays) refresh**. Grasp the "write the buffer, then refresh" step and you can pick up nearly any screen.

> ✦ **Key point:** character displays take character codes, pixel displays write a frame buffer; both commonly use **I²C on just 4 wires**. On a pixel display, remember to `display()` after drawing. Screen blank? Check the I²C address, the pull-ups, and whether you called init.
