## 核心讲义

> ✦ 本模块是一个**研究 / 原型级**项目,目标是「读懂原理、动手搭出可运行的样机」。所用器件均为现货爱好级物料,**不是医疗器械**。要做成受监管的商业产品,需替换为医疗级隔离器件并通过 IEC 60601 等认证(见 CB5)。

### 一张会「读」病人的床

把前六个模块学到的东西——电压电流、数字与模拟 I/O、微控制器、传感器、通信总线——组合起来,就能做一件真正有用的东西:一张能持续监测病人的**智能护理床**。它不是一台单独的仪器,而是一套**分布式的传感与判断系统**,悄悄嵌在床里,7×24 小时回答四个护理上真正要紧的问题:

1. **病人还在正常呼吸吗?** 呼吸暂停、呼吸频率骤变,是很多急症的早期信号。传统做法要绑胸带或贴电极,病人不适、依从性差。我们要做的是**完全非接触**地测(见 CB2)。
2. **多久没翻身了?是不是要离床?** 长期卧床病人如果几个小时不翻身,受压部位会缺血、坏死,形成**压疮(褥疮)**——这是护理质量最重要的指标之一。而失能、术后、认知障碍的病人擅自离床,极易跌倒(见 CB3)。
3. **床是不是湿了?** 尿失禁若不及时处理,会导致皮肤浸渍、感染,加速压疮。及时、准确的提醒能显著减轻护理负担(见 CB4)。
4. **万一断电、接触病人,安全吗?** 只要设备与人体有电气连接,患者安全(隔离、漏电流)就压倒一切;监护更不能因停电或转运而中断(见 CB5)。

这四个问题各自对应一类传感原理,也各自对应本课程前面学过的一块知识。把它们**融合**在一起,才是「智能」二字的真正含义:任何单一传感器都会误报,多路信息交叉印证,判断才可靠。

### 为什么要「分层」:大脑 + 感官

初学者常想:「用一块 Arduino 全搞定不行吗?」不行——因为这套系统里,有两类**性质完全不同**的任务:

- **实时、高频、时序敏感的采样**:比如每秒几百上千次读称重传感器、精确计时超声回波。这类任务要求「稳定的节拍」,一旦被别的事情打断就会失真。
- **重量级、不那么讲究实时的计算**:比如把原始波形做滤波、跑机器学习分类体位、渲染触摸屏界面、连 Wi-Fi 上传云端。这类任务偶尔卡顿几十毫秒无所谓,但很吃算力。

把这两类任务塞进同一颗芯片,会互相拖累:采样被界面刷新打断,或算法把 CPU 占满导致丢采样。工程上的标准解法是**分层 / 协处理器架构**——让专职的小芯片管实时,让强力的主控管思考。

### 主控(大脑):树莓派 5

系统的「大脑」是一台**树莓派 5**(4–8 GB 内存)+ 电源 + microSD 卡。它是一台完整的 Linux 计算机(参见 MC 模块对微控制器与单板机的区分),承担所有「需要思考」的活:

- **传感器数据融合**:把来自多个节点、多种传感器的数据流对齐、加权、交叉验证,得出「呼吸率 = X、体位 = 左侧卧、床面干燥」这样的结论。
- **边缘机器学习**:在本地跑轻量模型,区分「翻身」与「正常小动作」、从体重波动里分离心跳,而不必把敏感的健康数据传到云端。
- **用户界面**:驱动 7 英寸触摸屏,给床旁护理人员看实时曲线与报警。
- **网络通信**:通过以太网 / Wi-Fi 把状态与报警上报到护士站或手机。
- **存储与日志**:记录趋势数据,供医生回看。

树莓派跑 Linux、有充足内存和算力、生态丰富(Python、OpenCV、各种 ML 框架都现成),正适合这些活。但它**不适合**做硬实时采样——Linux 不是实时操作系统,它的调度会有不可预测的延迟。

### 传感器节点(感官):ESP32

真正贴着传感器、按固定节拍取数的,是一个或多个 **ESP32-DevKitC**(或 STM32 Nucleo 这类 Cortex-M 单片机)。它们是系统的「感官末梢」:

- **实时采样**:ESP32 是裸机 / RTOS 环境,能稳定地以固定频率读 ADC、计时,不受界面和网络干扰。
- **就近处理**:在节点上先做初步滤波、打包,减少要传给树莓派的数据量。
- **无线回传**:ESP32 自带 **Wi-Fi 与蓝牙(BLE)**,可以无线把数据发给树莓派——这意味着床脚、床垫、护栏上的传感器节点可以各自独立、少走线。

为什么用多个节点而不是一个?因为传感器分布在床的不同位置(四个床脚、床垫、湿度垫),就近各放一个节点,比把几十根模拟线从各个角落拉回一个中心更抗干扰、更好维护。这也是工业现场常见的**分布式采集**思路。

### 两层之间如何对话

大脑与感官之间需要一条通信链。常见几种选择,取舍不同:

| 方式 | 特点 | 适用 |
|---|---|---|
| **Wi-Fi(TCP/MQTT)** | 无线、带宽大、可跨房间 | 节点较远、要联云 |
| **BLE** | 低功耗、短距 | 电池供电的小节点 |
| **串口 UART**(见 BUS1) | 简单、可靠、有线 | 节点就在床内、走线方便 |
| **CAN 总线**(见 CB5) | 抗干扰、多节点、长距 | 多张床组网 |

无论走哪条链,节点与主控之间要约定一个**消息格式**:比如每帧包含「节点 ID + 时间戳 + 传感器类型 + 数值 + 校验」。时间戳尤其重要——要把多个节点的数据融合,必须知道它们各自「什么时候采的」,否则对不齐。

### 四类监测子系统与知识地图

| 子系统 | 监测什么 | 核心器件 | 用到的前置知识 | 章节 |
|---|---|---|---|---|
| 呼吸 / 心率 | 呼吸、心跳(BCG) | 4× 称重传感器 + HX711/ADS1232 | ADC(IO4)、信号调理 | CB2 |
| 翻身 / 体位 / 离床 | 姿态、压力分布 | FSR 压力阵列 + IMU | ADC、I²C(BUS2)、多路复用 | CB3 |
| 尿床 / 湿度 | 是否尿湿 | 湿度电极 + SHT40 | 分压(CF3)、I²C | CB4 |
| 供电 / 隔离 / 安全 | 患者安全、断电续航 | 隔离电源 + 数字隔离器 + 电池 | 电源、隔离、CAN | CB5 |

可以看到,这个项目几乎把整门课都用上了:**电阻分压**读湿度电极、**ADC** 读称重与压力、**PWM/GPIO** 驱动报警、**I²C** 接 IMU 与温湿度、**UART/CAN** 做节点通信、**电源与隔离**保障安全。这正是「项目模块」的意义——把散落的知识点缝合成一个真实系统。

### 端到端数据管线

把一次「监测到呼吸暂停并报警」拆开看,数据是这样流动的:

```text
称重传感器(mV 起伏)
  → 24 位 ADC 放大量化(HX711)
  → ESP32 采样、初滤波、打包
  → Wi-Fi/串口 传给树莓派
  → 树莓派滤波分离出呼吸波
  → 检测到 15 秒无呼吸起伏
  → 触发报警:屏幕 + 蜂鸣 + 上报护士站
```

每一环都可能出错:传感器松动、ADC 噪声、无线丢包、算法误判……一个可靠的系统要在每一环都考虑失败模式(见各章的「坑」)。

### BOM 哲学:先跑起来,再升级

本项目的物料清单(BOM)是**可直接采购**的:所有器件都是现货,可从 **Digi-Key、Mouser、亚马逊、Adafruit、SparkFun** 等平台买到。选型的核心思路是:

> **先用最便宜的、能跑起来的方案验证原理,再按需逐项升级。**

比如称重 ADC 先用几块钱的 HX711 把呼吸波跑通,确认整条链没问题,再换更贵、更低噪声的 ADS1232 去追求真正的心跳质量。这样每一分钱都花在「已被验证值得升级」的地方,避免一上来就买贵器件却发现思路错了。

清单也按「原型级 / 产品级」分层:原型级追求便宜好买、快速验证;产品级追求医疗合规、可靠耐用。**可打印的完整采购清单见 [BOM 页面](#/bom)**,那里按子系统分组、标注了入门套件、可选与产品级替换。

### 最小化「入门套件」

如果只想先花最少的钱搭一个能动的原型,买这些就够:

> 树莓派 5 · ESP32 · 4×50 kg 称重传感器 · HX711 · Velostat 薄膜 + 铜箔胶带(压力网格)· MPU-6050 · 电容式湿度传感器 · SHT40 · 面包板 + 跳线 + 5 V 电源。

用它你就能把四个子系统各跑通一个最小版本,建立起对整条链的直觉。之后再决定往哪个方向深挖、升级。

### 时间同步:融合的隐形前提

多节点系统里有一个容易被忽视、却决定成败的问题:**时间同步**。要把「床脚 A 的呼吸波」和「IMU 的翻身事件」融合判断,必须知道它们各自「什么时候发生」。可两个 ESP32 节点各有自己的晶振,时钟会缓慢跑偏(**时钟漂移**),几分钟后就可能差出几十毫秒——足以让本该对齐的事件错位。

常见的解法有几种,由简到繁:

- **主控盖章**:节点数据一到树莓派就打上「到达时间」戳。简单,但忽略了传输延迟的抖动,精度有限。
- **节点本地时间戳 + 周期对时**:节点自己给每个采样打时间戳,树莓派定期广播一个基准时间让各节点校准(类似简化版 NTP)。精度好很多。
- **硬件同步脉冲**:主控发一根共享的同步信号线到所有节点,大家以它为节拍。精度最高,但要多走一根线。

对护理床这种「秒级事件」的场景,第二种方案通常足够。理解「多节点必须对时」这件事本身,比具体用哪种更重要。

### 报警系统:比传感器更难的部分

真正的护理设备里,**报警系统往往比传感器本身更难做对**,以至于有专门的标准 **IEC 60601-1-8** 来规范它。核心矛盾是:

- **漏报(该响没响)是致命的**——呼吸暂停没报警,后果不堪设想;
- **误报(不该响乱响)会摧毁信任**——护士被误报折腾几次,就会调低灵敏度甚至关掉报警,系统随之失效。

好的报警设计要处理:**分级**(信息提示 / 警告 / 紧急,用不同声光区分)、**优先级**(多个报警同时发生时先报最要紧的)、**确认与静音**(护士能临时静音但系统记录在案)、**自检**(报警器自己坏了要能被发现——「报警系统失效」本身就是一种最高级报警)。这也是为什么本项目把「报警」当成一个需要认真设计的子系统,而不是「检测到异常就 `digitalWrite(BUZZER, HIGH)`」那么简单。

### 可靠性与冗余:假设每一环都会坏

医疗/护理设备的设计哲学是**「假设每个部件都会在最糟的时刻失效」**,并让系统在失效时进入**安全状态**、并且**让失效可被发现**。几个常用手段:

- **看门狗(watchdog)**:让 MCU 定时「喂狗」,一旦程序卡死不喂,看门狗自动复位芯片,防止死机后无声无息地停止监测。
- **心跳检测**:节点之间、节点与主控之间周期性发「我还活着」的心跳包。一旦某节点的心跳消失,主控立即知道「这路传感器掉线了」并报警——**沉默的传感器和正常的传感器必须能区分**。
- **冗余**:关键指标(如呼吸)可用两种独立原理交叉测量,一路失效还有另一路。
- **优雅降级**:某个子系统坏了,其余功能继续工作,而不是整机罢工。

「让失效可被发现」尤其重要:一个悄悄停止工作、却还显示着旧数据的监护仪,比一个明确报「我坏了」的监护仪危险得多。

### 数据与隐私:为什么倾向本地优先

健康数据是高度敏感的隐私。本项目倾向**边缘计算 / 本地优先**——尽量在树莓派本地完成融合与判断,只在必要时(如报警、趋势汇总)才上传,而非把原始生理数据一股脑传上云。这样既降低隐私风险与合规负担,也减少对网络的依赖(网断了核心监测不能停)。这正是 CB1 选「树莓派做本地 ML」而非「全靠云」的深层原因之一。

| 部分 | 大致占比 | 说明 |

### 动手:从零搭一个最小可用系统

不用一上来就四个子系统全上。按下面的顺序,你可以在桌面上一步步把整条链搭通,每一步都能验证:

1. **点亮主控**:给树莓派 5 烧好系统卡(参考 MC 模块 / GC2 的烧录流程),接屏、开机、连上 Wi-Fi。确认能跑 Python。
2. **单节点单传感器**:先只做呼吸——把 4 个称重传感器合成全桥、接 HX711、接到一个 ESP32。用现成库读出重量,串口打印。看到数字随你压床变化,第一环就通了。
3. **无线打通**:让 ESP32 通过 Wi-Fi(或串口)把读数发给树莓派;树莓派收到并画成曲线。这一步验证了「感官→大脑」的通信。
4. **加一路,验证融合**:再接一个 IMU(I²C)到 ESP32,把「重量起伏」和「朝向」两路一起发给树莓派。此时你已经有了最小的多传感器融合原型。
5. **报警闭环**:在树莓派上写个简单规则(如「15 秒无呼吸起伏就报警」),触发蜂鸣器 + 屏幕提示。至此,一条完整的「传感→融合→判断→报警」闭环跑通了。

之后再按需扩展压力阵列(CB3)、湿度(CB4)、供电与隔离(CB5)。**先把一条最短的链从头跑到尾,再往两边加宽**——这是搭复杂系统最稳的节奏,避免一次接一堆东西、出了问题无从下手。

### 常见问题排查(系统级)

| 现象 | 可能原因 | 排查方向 |
|---|---|---|
| 节点数据收不到 | Wi-Fi 没连上 / IP 变了 / 消息格式不对 | 查节点串口日志、固定 IP、核对帧格式 |
| 多路数据对不齐 | 没做时间同步 | 加时间戳、周期对时(见上文) |
| 树莓派偶发卡顿 | 界面/网络抢占了 CPU | 把实时采样交给节点,别塞给树莓派 |
| 某传感器悄悄失效却没发现 | 没做心跳检测 | 加节点心跳,掉线即报警 |
| 一动全是噪声 | 机械松动、共地/接地问题 | 稳固安装,检查共地(见 CB2/CB5) |

### 从桌面原型到床上部署

最后别忘了:这是一个要装到**真实的床**上的系统,机械集成和电子同样重要。称重传感器要稳固地垫在床脚下且不晃(否则 BCG 全是噪声,见 CB2);压力垫要平整地铺在床垫下且耐反复受压;湿度电极要贴合又可更换;所有走线要固定、不绊人、能随床架活动;外壳要能擦洗消毒。**一个在桌面上跑得完美、装到床上就因为晃动、走线、清洁问题全面翻车的原型,是常态而非例外**——从桌面到床上,是这个项目真正的「最后一公里」。

> ✦ **要点:** 智能护理床 = **树莓派做融合、ML 与界面 + 一个或多个 ESP32 节点做实时采样**,下挂呼吸、体位、湿度、供电四类子系统。分层的理由是「实时采样」与「重量计算」性质相反,硬塞进一颗芯片会互相拖累。先用现货爱好级物料按「先跑通再升级」的思路搭原型;走向产品化再谈医疗级与合规(CB5)。整个项目是前七个模块知识的一次总演练。
