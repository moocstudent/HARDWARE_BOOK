## 核心讲义

### 把掌机当成一台可编程的 Linux 设备

前面三章把它当游戏机用:选硬件、刷系统、跑模拟器。这一章把它当**电脑**用——看懂它的文件结构、远程登进去、理解它的输入系统,甚至在上面跑你自己写的程序。这才是「掌握硬件」与「只会玩」的真正分界线,也是本模块、乃至整门课的落点:从「读一个引脚」一路走到「给一台设备编程」。

### SD 卡目录布局

游戏卡(见 GC2 的双卡槽)的目录结构是高度约定俗成的。以 ArkOS 为例,ROM 按主机分文件夹存放:

```text
/roms/
  ├── snes/          SNES 游戏
  ├── nes/           NES 游戏
  ├── gba/           GBA 游戏
  ├── psx/           PS1 游戏(.chd / .zip)
  ├── arcade/        街机 ROM
  ├── ...
  └── bios/          各主机所需的 BIOS 文件
```

每个主机一个文件夹,文件夹名是固定的(前端靠它认主机)。往里放对应格式的 ROM,再在前端选「更新 / 刷新游戏列表」重新扫描,新游戏就出现了。有些固件还有 `roms/xxx/images` 之类的子目录放封面图。

**加游戏最朴素的方式**:关机、把游戏卡拔下来插到电脑上、拷文件、插回去。可行但麻烦(要频繁插拔卡,还得关机)。更优雅的方式是走网络——这就要先让掌机联网。

### 联网:Wi-Fi 适配器 + SSH

许多入门机(包括 R36S)**没有内置 Wi-Fi**(回顾 GC1 的避坑),要外接一个 **USB-C Wi-Fi 适配器**。选适配器的关键是**芯片要被 Linux 内核支持**——常见受支持的芯片如 **RTL8188** 系列,买之前查该固件社区的兼容列表,别买回来发现驱动没有、认不出。

联网之后,因为它本质是一台 Linux 机器,你就能用标准的 Linux 远程工具管理它,像操作一台服务器:

```bash
# 从电脑通过 SSH 登入掌机(默认用户名/口令见固件文档,
# ArkOS 常见为 ark / ark)
ssh ark@192.168.1.50

# 登进去后你就有了一个完整的 Linux 终端:
ls /roms/snes            # 看游戏目录
df -h                    # 看 SD 卡空间
top                      # 看 CPU/内存占用
```

传游戏也不用拔卡了,直接用 **scp** 从电脑推过去:

```bash
scp Chrono_Trigger.zip ark@192.168.1.50:/roms/snes/
```

很多固件默认还开了**网络共享(Samba)**:在电脑的文件管理器地址栏输入 `\\掌机IP`(Windows)或 `smb://掌机IP`(Mac/Linux),就能像访问网络硬盘一样直接看到 `roms` 文件夹,拖拽拷贝,最省事。

### 输入:手柄如何变成按键

要理解「怎么给它编程读手柄」,得先理解 Linux 是怎么看待手柄的。在 Linux 里,一切设备皆文件——掌机的实体按键、十字键、摇杆,通过内核的 **evdev(event device)**子系统,被抽象成一个输入设备文件,通常是 `/dev/input/event0` 之类。

这个设备会不断上报**输入事件**:「某个键被按下」「某个键松开」「摇杆 X 轴到了某个数值」。每个事件带有类型(按键还是轴)、编码(哪个键/轴)、数值。RetroArch 和前端就是读取这些 evdev 事件,再按你设置的**映射表**把物理按键对应到虚拟手柄的按键(比如把实体的 X 键映射成模拟器里的「A」)。

理解这一点,你就明白:「重新映射按键」改的是这层映射表;而你自己写程序时,只要读同一个 evdev 设备,就能拿到手柄输入。

### 跑你自己的程序

既然是一台 Linux 机器、通常预装了 Python,你完全可以把它当一台小电脑,写自己的应用。下面是一个最小示例:直接读手柄事件(用 `evdev` 库):

```python
# 读手柄事件的最小示例
from evdev import InputDevice
dev = InputDevice('/dev/input/event0')
print('listening on', dev.name)
for event in dev.read_loop():
    if event.type == 1 and event.value == 1:   # type 1 = 按键, value 1 = 按下
        print('key', event.code, 'pressed')
```

有了「读输入 + 完整 Linux + Python」,你能做的事就多了:

- **自动化脚本**:批量整理 ROM、重命名、按规则归类、自动抓封面图。
- **自制启动器 / 菜单**:写一个自己的界面,启动游戏或工具。
- **轻量小游戏 / 应用**:用 Python(如 Pygame)在掌机上跑自己写的小游戏、电子书阅读器、音乐播放器。
- **硬件改装**:部分机型在主板上引出了调试脚 / GPIO,动手能力强的可以接传感器、灯,做硬件 mod(这就把整门课的 GPIO、I²C 等又用上了)。

到这一步,这台掌机**不再只是个玩具,而是一台你能完全掌控、能编程的手持 Linux 计算机**。你会读它的规格(GC1)、会换它的系统(GC2)、懂它的软件栈(GC3)、能登进去改它、给它写程序(GC4)——这正是「掌握硬件」的完整含义:不被设备的预设用途所限,而是理解它、驾驭它、扩展它。

### 现实中的坑

- **默认口令与安全**:很多固件默认 SSH 口令是公开的(如 ark/ark)。若掌机常连公共 Wi-Fi,考虑改口令。
- **只读系统**:JELOS 等只读固件下,系统分区不可写,想持久保存自己的脚本要放在可写的用户/游戏分区。
- **Python 版本与库**:预装的 Python 版本可能较旧、缺库。装库可能需要联网,或在电脑上交叉准备。
- **别乱动系统文件**:改配置前先备份;实在弄乱了,大不了重烧系统卡(游戏卡不受影响,见 GC2)。

### 传文件的几种方式,横向对比

「怎么把游戏弄进去」有好几条路,按方便程度和门槛对比:

| 方式 | 需要联网? | 门槛 | 适合 |
|---|---|---|---|
| 拔卡插电脑拷 | 否 | 最低 | 一次性大批量、没 Wi-Fi 时 |
| Samba 网络共享 | 是 | 低 | 日常拖拽,最省事 |
| scp / SFTP | 是 | 中 | 命令行党、脚本自动化 |
| USB 直连(部分机型) | 否 | 中 | 有 USB-OTG 且固件支持 |

新手从「拔卡拷」起步最稳,联网后转向 Samba 最舒服,想自动化再上 scp。理解「这是台 Linux 机器,标准的 Linux 文件传输手段都能用」这一点,比记具体命令更重要。

### 存档:最该备份的东西

游戏可以重下,**存档丢了就没了**——几十小时的进度可能一夜蒸发。存档通常在游戏卡/数据分区的 `saves`(或各模拟器自己的目录)里。养成习惯:定期把 `saves` 目录整个拷出来备份。换固件、重烧卡前尤其要先备份存档(回顾 GC2)。有了 SSH/Samba,备份可以一条命令搞定:

```bash
scp -r ark@192.168.1.50:/roms/saves ./backup_saves
```

### 联网还能做什么:成就与联机

一旦掌机能上网,除了传文件,还解锁一些玩法:

- **RetroAchievements(复古成就)**:给老游戏加上现代的成就系统,通关、隐藏要素都能解锁徽章,给重温老游戏加了个目标感。需要联网登录。
- **Netplay(联机对战)**:RetroArch 支持通过网络和别人玩同一个模拟游戏(如街机对战)。延迟敏感,但原理有趣。
- **在线刮削**:联网后前端能自动下载封面和元数据(见 GC3),游戏库瞬间变漂亮。

这些都是「因为它本质是台联网的 Linux 电脑」才有的能力,再次印证本模块「把它当电脑用」的主旨。

### 不只是模拟器:跑原生程序与移植游戏

因为底层是标准 ARM Linux,很多**原生程序**能直接跑或稍加移植就能跑:

- **原生移植游戏(ports)**:社区把很多经典 PC 游戏移植到了这些掌机上原生运行——如 *Doom*、*Quake*、*OpenTTD*、*Cave Story*、各种解谜/独立游戏。它们不经模拟、直接编译成 ARM 运行,往往比模拟更流畅。
- **实用工具**:文件管理器、音乐/视频播放器、电子书阅读器,都有人做。
- **你自己写的东西**:结合前面 evdev 读手柄的例子,用 Python + Pygame 就能在掌机上跑自己写的小游戏或工具。

「能跑原生 ARM Linux 程序」这件事,把掌机从「专用模拟器盒子」彻底变成了「通用手持电脑」。

### 进阶:交叉编译与开发

想在掌机上跑更重的自研程序,直接在机上编译又慢又缺工具,常见做法是**交叉编译**:在性能强的 PC 上,用针对 ARM 的工具链把程序编译成掌机能跑的可执行文件,再传过去运行。这正是嵌入式开发的标准流程——回顾整门课,从给 Arduino「烧录」程序,到给这台 ARM Linux 掌机「交叉编译并部署」程序,你已经走过了从最小单片机到完整 Linux 设备的全谱系。

### 界面主题与个性化

既然是开源固件,界面也能改。EmulationStation 支持**主题(theme)**,社区做了大量精美主题,换一个就能让你的掌机界面焕然一新;还能调整启动画面、音效、收藏夹、游戏分类。这些虽不改变性能,却是「这台设备真正属于我、被我掌控」的直观体现——从被动接受出厂设定,到主动定制每一处,正是「用户」到「主人」的转变。

### 动手:第一次 SSH 登进你的掌机

把「它是台 Linux 电脑」这句话变成亲手体验:

1. **联网**:插上受支持的 USB-C Wi-Fi 适配器,在系统的网络设置里连上和你电脑同一个 Wi-Fi。
2. **查 IP**:在掌机的网络信息里看它的 IP(如 192.168.1.50),或在路由器后台找。
3. **确认 SSH 开着**:多数固件(ArkOS 等)默认开了 SSH;若没开,在设置里打开。
4. **从电脑登入**:

```bash
ssh ark@192.168.1.50      # 默认用户/口令见固件文档,ArkOS 常见 ark/ark
```

5. **验证你在一台 Linux 上**:

```bash
uname -a                  # 看内核与架构(ARM)
ls /roms                  # 看游戏目录
free -h                   # 看内存
```

6. **推个游戏试试**(免拔卡):

```bash
scp Sonic.zip ark@192.168.1.50:/roms/genesis/
```

回掌机前端「更新游戏列表」,新游戏就出现了。这一刻,你就完成了从「玩游戏机」到「远程管理一台 Linux 设备」的跨越。

进阶:传一个 `hello.py`(比如前文读手柄的例子)上去,在 SSH 里 `python3 hello.py` 跑起来——你就在掌机上跑了自己写的程序。

### 常见问题排查(对接)

| 现象 | 可能原因 | 排查方向 |
|---|---|---|
| Wi-Fi 连不上 | 适配器芯片不被支持 | 换 RTL8188 等受支持芯片(见 GC1/GC2) |
| SSH 连不上 | SSH 没开 / IP 变了 / 不同网段 | 开 SSH、查最新 IP、确认同一 Wi-Fi |
| 登进去密码不对 | 用了错的默认口令 | 查该固件文档的默认用户/口令 |
| 拷进去的游戏不显示 | 目录/格式不对、没刷新列表 | 放对 `/roms/<主机>/`、更新游戏列表 |
| 脚本存了重启就没 | 系统只读、存到了只读区 | 存到可写的用户/游戏分区(见 GC2) |
| 手柄读不到事件 | event 设备号不对 | `ls /dev/input/` 找对的 eventN、装 evdev 库 |

> ✦ **要点:** 把掌机当一台可编程的 Linux 设备:**SD 目录**按主机放 ROM/BIOS;插 **USB Wi-Fi**、用 **SSH / scp / Samba** 远程管理与传文件;手柄经内核 **evdev** 抽象成事件、按映射表变按键;最终用 **Python 等在上面跑你自己的代码**。走完这几步,你就从「用户」变成了「能改能编的开发者」——这是本模块的终点,也是整门课从「读引脚」到「写程序」的收束。
