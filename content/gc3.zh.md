## 核心讲义

### 从按键到画面,中间隔着三层

你选中一个游戏、按下 A 键,屏幕上就出现了童年的画面。这背后不是「一个模拟器」在干活,而是**一整层软件栈**在协作。看不懂这层栈,遇到「进不去游戏」「卡顿」「没声音」就只能干瞪眼;看懂了,你就知道每个问题该去哪一层调。

```text
你(按键)
 → EmulationStation(前端:浏览、选择、启动游戏)
 → RetroArch(统一的运行环境 / 调度器)
 → libretro 内核(真正在模拟某台主机的那段代码)
 → ROM(游戏数据) + BIOS(主机固件)
 → 画面 / 声音 / 手柄反馈
```

三个关键层次——**前端、RetroArch、内核**——各管一段。下面逐层拆开。

### 前端:EmulationStation

**EmulationStation(ES)**是你开机看到的那个漂亮界面。它按主机分类(NES、SNES、PS1……),展示游戏封面、抓取媒体信息、让你用手柄浏览、选择、启动游戏。

关键要理解:**ES 本身不模拟任何东西**,它只是一个「启动器 / 门面」。它的工作是「让你舒服地找到并选中一个游戏」,选好之后,它把「哪台主机 + 哪个 ROM 文件」这个信息交给下一层去真正运行。所以 ES 的问题多是「界面、分类、封面图」层面的;游戏本身跑得好不好,不归它管。

不同固件可能用不同前端(有的用 ES,有的用自研界面),但「前端只负责选、不负责跑」这个分工是一致的。

### RetroArch 与 libretro 内核

选好游戏后,真正干活的是 **RetroArch**。但注意——**RetroArch 自己也不模拟主机**!它是一个**统一的前端 / 运行框架**,提供了菜单、存档、手柄映射、着色器、录制等一大堆通用功能,然后加载一个个**libretro 内核(core)**来做真正的模拟。

**libretro** 是一套统一的接口规范。每个 **内核** 就是「某一台主机的模拟器」,被打包成符合 libretro 接口的插件。这样设计的妙处是:所有内核共享 RetroArch 的通用功能(存档、金手指、着色器对所有内核都一样能用),而 RetroArch 不用关心每个内核内部怎么模拟。

| 主机 | 常用内核(举例) | 备注 |
|---|---|---|
| NES | FCEUmm / Nestopia | Nestopia 更准,FCEUmm 更快 |
| SNES | Snes9x / snes9x2010 | 后者为慢机器优化 |
| GBA | mGBA / gpSP | mGBA 准,gpSP 快 |
| PS1 | PCSX-ReARMed | **专为 ARM 优化,入门机首选** |
| N64 | Mupen64Plus-Next | 吃性能,入门机勉强 |
| 街机 | FBNeo / MAME | 需匹配 ROM 版本 |

**同一台主机常有多个内核可选**,这是关键知识点:有的内核**准确**(高度还原原机行为,但吃性能),有的**快**(为弱机优化,牺牲一点兼容性/精度)。在 R36S 这类入门机(见 GC1)上,PS1 用**为 ARM 优化的 PCSX-ReARMed** 就能流畅,换个「准确但重」的内核可能就卡。**「同样的机器、同样的游戏,选对内核就能从卡变流畅」——这正是理解这层栈的实用价值。**

### BIOS、ROM 与文件放置

模拟器要跑起来,需要两类文件:

- **ROM**:游戏本身的数据文件(卡带/光盘的镜像)。按主机放进对应目录(见 GC4 的 SD 卡布局),如 `/roms/psx/`、`/roms/snes/`。前端扫描这些目录来生成游戏列表。
- **BIOS**:有些主机(PS1、世嘉土星、部分 GBA、NDS 等)需要**原机的固件文件**才能启动——因为模拟器要模拟的不只是 CPU,还有主机开机时运行的那段固件。BIOS 文件放在指定的 `bios` 目录,且**文件名和校验和(MD5)必须精确匹配**,差一个字节都不行。

> ⚠ **「游戏进不去、黑屏、报错」的头号原因就是缺 BIOS 或 BIOS 不对。** 遇到某主机所有游戏都启动失败,先查它是不是需要 BIOS、你放的 BIOS 文件名和校验和对不对。

常见文件格式:`.zip`(压缩的 ROM,多数内核可直接读)、`.chd`(压缩光盘镜像,PS1/土星等大容量光盘游戏首选,又小又能直接跑)。

> ⚠ **版权提醒**:请只使用你**合法拥有**的游戏与固件的备份。ROM 与 BIOS 的版权归原厂所有;本课程只讲技术原理,**不提供、不链接任何 ROM/BIOS 文件**。

### 存档、金手指、着色器:RetroArch 的通用能力

因为这些功能做在 RetroArch 层(而非某个内核里),所以它们对**几乎所有内核**都通用——这正是统一框架的价值:

- **即时存档(Save State)**:随时把整个模拟器的运行状态(内存、寄存器、画面)完整保存成一个文件,之后瞬间恢复。它不依赖游戏自带的存档点,可以在任何一帧存档——想在 Boss 战前存一下?随时。
- **金手指(Cheats)**:加载作弊码,修改游戏内存实现无敌、无限金钱等。
- **着色器(Shader)**:用 GPU 滤镜还原老屏幕质感——CRT 扫描线、荫罩、LCD 网格、曲面等,让画面更有「当年味」。**但着色器吃 GPU**,在 R36S 这类入门机上要权衡:开重着色器可能掉帧,得选轻量的或不开。
- **快进 / 倒带**:加速跑图,或「倒带」几秒重来(对付手残很有用)。
- **每游戏配置**:可为单个游戏单独设内核、按键、着色器,不影响其它游戏。

### 排障心法:分层定位

理解了三层栈,排障就有了章法:

- **某个主机所有游戏都进不去** → 多半是 **BIOS** 问题或该主机的内核没装好(内核/BIOS 层)。
- **能进但卡顿/掉帧** → 换个**更快的内核**、关掉重着色器、或这台机器就是带不动这代主机(GC1 的性能上限)。
- **界面里看不到游戏** → **前端**没扫描到,检查 ROM 放对目录没、格式对不对、更新一下游戏列表。
- **没声音、按键不对** → RetroArch 的音频/输入设置或手柄映射(见 GC4)。

### 模拟到底在「模拟」什么

一个内核跑起来,内部到底在干什么?它要在软件里**假装自己是那台老主机的全部硬件**:模拟它的 CPU(逐条读取游戏的机器指令并执行等效操作)、模拟它的显示芯片(把游戏写进「显存」的数据画成画面)、模拟它的声音芯片、手柄接口、卡带/光盘读取……游戏本身是为真实硬件写的,它不知道自己跑在模拟器里,内核要骗过它。

翻译 CPU 指令有两种做法,性能天差地别:

- **解释执行(Interpreter)**:逐条读原机指令、逐条翻译执行。简单、准确、好移植,但慢——每条指令都要绕一大圈。
- **动态重编译(Dynarec / JIT)**:把一段原机指令**批量翻译成宿主机(ARM)的原生指令**并缓存起来,下次直接跑翻译好的。快得多,但复杂、且要针对宿主 CPU 架构专门写。

这就是为什么「为 ARM 优化的内核」(如 PS1 的 PCSX-ReARMed)在掌机上快——它有针对 ARM 的动态重编译。理解这一层,你就明白 GC1 说的「选对内核能把卡变流畅」背后的原理:同一个游戏,解释执行的内核可能卡,带 ARM dynarec 的内核就流畅。

### 为什么越晚的主机指数级难模拟

GC1 说「越晚的主机越吃算力」,这里补上原理。难度暴涨来自几方面叠加:

- **CPU 更快更复杂**:要模拟的指令更多、更花哨;
- **多芯片并行**:PS2、土星那种有好几颗协处理器同时工作,模拟器要同步模拟多颗芯片,难上加难;
- **3D 图形**:早期 2D 只需画像素,后期 3D 要模拟一整套 GPU 管线(几何、光栅、纹理),对宿主 GPU 要求高;
- **精度要求**:很多游戏依赖原机的时序细节,模拟不准就出 bug,而追求精度又更慢。

所以从 PS1 到 PS2 不是「难一点」,而是「难一个数量级」——这解释了 GC1 那张能力表里为什么在 PS1 和 PS2 之间有一道清晰的鸿沟。

### 帧率、音画同步与超频

模拟不仅要「算得出」,还要「算得及时」。老主机大多 **50/60 Hz** 刷新,内核必须在每 1/60 秒内算完一帧,否则**掉帧**(卡顿)。音频更娇气:声音必须连续不断地喂给声卡,一旦算力不够、缓冲欠载,就会出现「爆音、卡顿、变调」。所以很多时候「一卡就破音」其实是算力不够的信号。

RetroArch 提供一些应对手段:**帧率跳过(frameskip)**(实在算不过来就跳过画面渲染保住游戏逻辑与声音)、**音频同步**、以及部分内核的**超频/精度选项**(如 N64 内核可下调精度换速度)。这些都是「用画质/精度换流畅」的旋钮——在入门机上,学会调这些旋钮,往往能把「几乎能玩」变成「顺畅能玩」。

### 让游戏被正确识别:刮削与游戏列表

把 ROM 拷进目录后,前端是怎么显示出漂亮封面和游戏信息的?靠**刮削(scraping)**:前端根据文件名去在线或本地数据库匹配,下载封面图、简介、发行年份等元数据,生成一个**游戏列表(gamelist)**文件。

所以有时「游戏在,但没封面/名字乱码」,是刮削没做或没匹配上——不是游戏本身的问题。文件名规范能大幅提高刮削成功率。这属于**前端层**的事,和游戏跑不跑得动无关,印证了 GC3 开头「分层定位问题」的思路。

### RetroArch 内核 vs 独立模拟器

不是所有模拟器都做成了 libretro 内核。有些复杂主机(如某些 PSP、PS2 模拟器)有**独立(standalone)**版本,功能更全、性能调校更专。很多固件会「简单主机用 RetroArch 内核、复杂主机用独立模拟器」混搭。理解这点,你就不会困惑「为什么这台机器上 PSP 不走 RetroArch」——分层的思想不变,只是某一层换了个更专的实现。

### 快捷键:掌机上的隐藏操作

掌机没有键盘,RetroArch 的菜单、存档、退出全靠**组合快捷键**——通常是一个 **Hotkey 键(常映射到 SELECT 或专门的功能键)**加另一个键。比如「Hotkey + X」呼出菜单、「Hotkey + R1」即时存档、「Hotkey + Start」退出游戏(具体看固件)。记住你固件的快捷键表,是流畅使用的前提。找不到菜单、退不出游戏,几乎都是没记快捷键——这也是新手最常见的困惑之一。

### 动手:把一个卡顿的游戏调流畅

这是最能体现「看懂栈」价值的实操。假设某个 PS1 或 N64 游戏卡顿,按下面顺序逐项试(从代价最小到最大):

1. **确认瓶颈**:开 RetroArch 的**帧率显示**,看是稳定 60、还是掉到 40。掉帧才是性能问题,不是别的。
2. **换更快的内核**:进 RetroArch 菜单换一个为 ARM 优化的快内核(PS1 用 PCSX-ReARMed)。这是最常见、最有效的一招。
3. **关掉重着色器**:如果开了 CRT 等着色器,先关掉——它们很吃 GPU。
4. **调内核选项**:很多内核有「速度 vs 精度」的选项(如降低内部分辨率、关某些增强、开动态重编译),往「快」的方向调。
5. **开 frameskip**:实在算不过来,允许跳帧保住流畅度与声音。
6. **认清上限**:若都试过还卡,可能就是这台机器带不动这代主机(回到 GC1 的性能上限)——这不是设置问题,是硬件天花板。

走完这套流程,你会真切体会到「同一台机器、同一个游戏,选对内核和设置就能从卡变流畅」——这正是理解三层栈的实用回报。

### 常见问题排查(模拟)

| 现象 | 在哪一层 | 排查方向 |
|---|---|---|
| 某主机所有游戏都进不去 | 内核 / BIOS | 查该主机是否需要 BIOS、文件名校验和对不对 |
| 能进但卡顿掉帧 | 内核 / 性能 | 换快内核、关着色器、调选项、开 frameskip |
| 游戏列表里看不到游戏 | 前端 | 放对目录、格式对、更新游戏列表 |
| 没封面 / 名字乱码 | 前端 | 重新刮削、规范文件名 |
| 没声音 / 破音 | RetroArch / 性能 | 查音频设置;破音多为算力不足 |
| 按键不对 / 退不出 | RetroArch 输入 | 重设按键映射、记住快捷键(Hotkey 组合) |

> ✦ **要点:** 模拟栈是三层:**前端(ES)负责选游戏 → RetroArch 负责调度与通用功能 → libretro 内核负责真正模拟某台主机 → 靠 ROM + BIOS 提供数据**。关键实用知识:**同一主机有多个内核,选对(尤其为 ARM 优化的快内核)能把卡变流畅**;**缺/错 BIOS 是进不去游戏的头号原因**;存档、金手指、着色器是 RetroArch 的通用能力。这套分层在任何 Linux 掌机上都通用。
