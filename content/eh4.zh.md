## 核心讲义

### 一句话先立住

**RS-485 / Modbus 解决「设备和设备怎么连」;MQTT / HTTPS 解决「事件怎么进 IT/ERP」。**  
中间必须有**边缘网关**做翻译——ERP 从不直接「听」485 总线上的寄存器。

这是整门课通信知识的工业终章:UART(BUS1)的差分多站版 + ESP32 Wi-Fi(MC2)的消息发布 + EH2/EH3 的业务事件。

### 四层对接栈

```
┌─────────────────────────────────────────────┐
│ ④ ERP / MES 事务层                           │
│    报工、收货、停机码、库存移动               │
│    入口: REST API / 中间件适配器              │
├─────────────────────────────────────────────┤
│ ③ 消息 / 集成层                              │
│    MQTT Broker、HTTPS 回调、重试、死信       │
├─────────────────────────────────────────────┤
│ ② 边缘网关层                                 │
│    协议转换 · 补主数据 · 校验 · 断网缓存     │
├─────────────────────────────────────────────┤
│ ① 现场总线 / 设备层                          │
│    RS-485 + Modbus RTU、UART 枪、GPIO 安灯   │
└─────────────────────────────────────────────┘
```

### ① 现场层:RS-485 + Modbus

| 点 | 说明 |
|---|---|
| **RS-485** | 物理层:差分、可多点、抗干扰、百米级——对照 UART,但 A/B 双绞 + 终端电阻 |
| **Modbus RTU** | 应用层:主站轮询从站,读保持寄存器/线圈(产量、运行位…) |
| **不直接给 ERP** | 帧里只有地址+寄存器值,没有物料号、工单号、会计期间 |

典型接线:

```
[计数模块]──┐
[电表]──────┼── RS-485 ──► [边缘网关 = Modbus 主站]
[安灯控制器]┘
```

实验可用两块 ESP32 + 两片 MAX485:一块模拟从站(把「产量」放进寄存器),一块当主站轮询。

### ② 边缘网关四件事

1. **轮询** Modbus:`读从站 3 寄存器 40001–40004`;
2. **映射**:寄存器 + 工位配置表 → `{ wo, op, goodQty }`;
3. **校验**:工单是否下达、数量是否离谱、时钟是否同步;
4. **发布**:组装 JSON,`MQTT publish` 或 `HTTPS POST`;失败进本地队列。

没有这一层,要么 ERP 被寄存器洪水打挂,要么现场断网时整条线「失忆」。

### ③ MQTT 主题与载荷

```
Topic:  plant/{plantId}/line/{lineId}/confirm
QoS:    1(至少一次)  →  ERP 侧必须幂等
Retain: false        →  报工事件不要 retain
```

```json
{
  "eventId": "gw01-20260723-153012-0007",
  "plant": "1000",
  "wo": "WO-5521",
  "op": "0020",
  "goodQty": 12,
  "scrapQty": 0,
  "source": { "bus": "modbus-rtu", "slave": 3, "reg": 40001 },
  "ts": "2026-07-23T15:30:12+08:00"
}
```

**ERP 怎么接?**

- 路径 A:MES/ERP 自带 MQTT 适配器 → 内部报工 API;
- 路径 B:中间件订阅 MQTT → 转 REST/IDoc(传统套装更常见);
- 路径 C:网关直接 `HTTPS POST /api/confirmations`(枪机/平板更常见)。

MQTT 擅长大量、异步、弱网;REST 擅长一问一答、强校验即时返回。工厂里两者经常并存——扫码收货走 HTTPS,高速计件走 MQTT。

### ④ ERP 事务层:收到 ≠ 已入账

1. 用 `eventId` **幂等**(重复投递不双记);
2. 校验工单/物料主数据;
3. 调用标准事务(报工、收货、停机单…);
4. **回执**:成功 ACK;失败进死信并告警——不能静默丢。

### 三种对照情景

| 情景 | 现场侧 | IT 侧 | 典型用途 |
|---|---|---|---|
| **A 计件报工** | PLC↔网关 **RS-485/Modbus** | 网关→**MQTT**→ERP | 高速产线 |
| **B 手持收货** | RF 枪 **USB/Wi-Fi** | App **HTTPS REST**→ERP | 仓库收发货 |
| **C 机床** | CNC **OPC-UA** | 网关→MQTT 或 OPC 桥 | 利用率、报警 |

OPC-UA 多在车间以太网,不是 485;但「先现场协议、再网关、再 IT 消息」的分层不变。

### 迷你实验(建议半天)

1. 从站 ESP32:Modbus 寄存器 `40001` = 模拟产量计数(按钮累加);
2. 主站/网关 ESP32:每秒轮询,差值 > 0 时组 JSON;
3. 发布到本地 Mosquitto,或 `POST` 到一个返回 200 的假 API;
4. 拔掉 Wi-Fi,确认事件进本地队列;恢复后按 `eventId` 刷出且不双发。

对照 ERP_BOOK HW4 的沙盘与主题约定,两边用同一份 JSON 字段名,联调成本最低。
